Panel 2 — why federation needs a key in that user entry

Zoom out from one island to the mesh. Two sovereign islands run today. The same human can be on both — and that is where Blocker 2 stops being a detail and becomes the whole game.

live today design / not built (the no-regrets move) the problem
→ #1680 auth on the bus. If the entry holds a key, a client proves who it is by signing ("I'm nick, here's my key"). Today mosquitto is allow_anonymous — the WSS/REST front door is locked but the bus behind it isn't. The key closes that.
→ #1569 sybil resistance. If identity is a key and minting one on an island costs something, you can't cheaply spin up a thousand fake principals. Federation trust rests on that cost, not on personhood.
Already live: the directory. Each island already lists the other (seed-peers). The mesh exists — what it's missing is a shared way to name who, which is exactly the key.

The through-line, Panel 1 → 2: "what's stored in a user entry?" looked like a small ChatServer detail. Zoomed out, the answer a public key is the same primitive that closes the bus's open door (#1680) and makes sybil-resistance possible (#1569). That's why the memory calls key-as-identity the no-regrets move — one decision, three payoffs.
