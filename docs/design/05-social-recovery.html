Social recovery guardian approval quorum · v2

The recovery hatch Design 04 left as an open fork. Decision (Nick, 2026-07-10): self-custody social recovery. After an adversarial design temper (four lenses, 2026-07-10) the mechanism moved from passive key-splitting (Shamir) to a guardian approval quorum (Nick, 2026-07-11): to recover, k of your n chosen guardians each sign a fresh approval of that specific recovery request. No secret is ever split, stored, or delivered; no island is a recovery authority. Status: design v2 — post-temper, for the gateway build. 1. Headline

Recovery is an account-takeover primitive by construction: whoever can re-bind a passkey owns the account. Social recovery makes the authority to do that a live quorum of friends you chose — never Google, never the island. To recover, the client assembles k fresh signatures from your guardians over this request (the new passkey it is installing), then waits out a time-locked veto. The gateway stores only n opaque approver public keys and a threshold; it verifies signatures and never holds anything that can recover an account alone. The property that makes this work where passive key-splitting failed: recovery cannot happen in silence — it requires actively asking k of your friends, any one of whom can warn you.

2. What's decided (and what this is not)

QuestionDecision
Recovery authorityGuardian approval quorum: k-of-n guardians, each holding their own Ed25519 approver keypair, sign a fresh per-request approval. No secret is split or reconstructed.
TimingTime-locked veto: a validated recovery opens a pending state; existing devices are notified and can cancel during the window before it finalizes.
Guardian contactOut-of-band: the recovering user asks friends directly (the request + approvals move over channels they already trust). The island is never in the guardian path.
Solo / no-friendsPrinted fallback = a 1-of-1 policy: the user holds one approver key and prints it. A bearer secret with a named loss profile (§4).
Out of scope. Not moving auth onto the bus (#1680); not touching sovereign message signing (#1816); not the password path (/register//login, separable per Design 04 §5). One new capability: re-bind a passkey to an existing account when a quorum of its guardians approves.

3. Temper outcome — why v2 differs from v1

v1 (Shamir k-of-n sharing of a recovery secret R) went through a four-lens adversarial temper. It surfaced five CRITICALs that were not independent bugs but symptoms of one root: a passive, reconstructable secret. Switching to a live approval quorum dissolves the root.

The five CRITICALs and how v2 answers each
v1 criticalv2 resolution
C1 — the recovery signature signed a bare nonce, so it didn't bind the passkey being installed (envelope-vs-payload, the validate_origin class)Fixed by design + binding: every guardian signature is over domain-separated bytes that include the new credential — see §5.
C2 — finalize/cancel was observe-then-write; the owner's veto could lose the race (TOCTOU)Fixed: finalize is a single guarded UPDATE with the deadline folded into the WHERE, arbitrated by rowcount — §7.
C3 — "encrypt each share to the guardian's identity key" named a key that does not exist in this repo (all account keys are signing keys)Dissolved: nothing is encrypted or delivered. Guardians sign; signatures need only their public key.
C4 — in-network delivery made the island the guardian-key distributor → MITM harvests shares → self-custody falseDissolved: the island is never in the guardian path; there is no secret to harvest.
C5 — the veto was inert in the states recovery exists for (offline/unwatching owner); it was the only defense against guardian collusionDowngraded from sole defense to backstop: a quorum attack is non-silent (needs k friends actively asked) — the friend-grapevine is a push-independent alert, and the veto is now a second line, not the only one.

The HIGH/MEDIUM findings are folded into §5–§8 (signature binding, guarded-UPDATE finalize, reclaim-on-expiry write-once deadline, rotation-through-veto, finalize re-asserts account standing, single-door credential enrollment, constant-shape errors, distinct-approver enforcement). The temper was lens-diverse on strong models, not full cross-family — a calibrated pass for a 0-user, undeployed design, not a substitute for the code /cage-match the gateway PR still owes.

4. Threat model

Approver keys needn't belong to aiko accounts, so the social graph is opaque (keys, not identities) — a named, reduced exposure, not zero.

5. Primitives

Approver keys

A guardian holds an Ed25519 approver keypair, generated in-app. The public key is registered with the account's recovery policy; the private key stays with the guardian (their device, or exported however they like). Approver keys are independent of any aiko account — a guardian need not be an aiko user, and a guardian deleting their aiko account does not drop the owner below k (a v1-Shamir failure that simply does not exist here).

The signed request — binding is the whole point (fixes C1)

Each guardian signs domain-separated bytes that commit to the specific new credential, never a bare nonce: 
approval_bytes = "aikochat:recover:v1"           (domain tag, length-prefixed)
              ‖ server_nonce                     (single-use, issued at recover/start)
              ‖ account_id
              ‖ new_credential_id
              ‖ new_public_key                   (the passkey being installed)
The gateway recomputes approval_bytes from the attestation it just verified and checks each guardian signature against a stored approver pubkey. So an approval authorizes this recovery installing this credential — a captured approval cannot be replayed to install a different one. The domain tag also makes an approval structurally un-replayable into the message-signing path (signing.py's DOMAIN_TAG is a distinct string), closing the cross-protocol reuse the temper flagged.

Reuse, not reinvention

Signature verification reuses domain/signing.py's multibase Ed25519 decode (the same z… Multikey format as SigningKey.pubkey). The single-use server nonce reuses the passkey_service.consume_challenge guarded-UPDATE pattern. The staged credential is enrolled through the same passkey_service door as normal registration (single-door — no bespoke insert, so credential_id uniqueness and clone-detection can't drift).

Contrast with SigningKey (#1816), deliberately. The signing roster stores a pubkey the gateway never verifies against (a carrier, so per-user non-unique, collisions kept as evidence). Approver pubkeys are the mirror image: the gateway does verify against them, they are registered authenticated, and a quorum of them is authoritative-for-takeover (not for identity). Same table shape, opposite trust posture — stated so a reviewer doesn't pattern-match the carrier invariant onto it.

6. The flows

Enroll / rotate (authenticated)

  1. App collects n guardian approver pubkeys (each guardian generates a keypair; the owner's app gathers the public halves out-of-band) and a threshold k.
  2. POST /v1/auth/recovery/enroll (session-authed) {approver_pubkeys[], k}.
    • First enrollment (no existing policy): stored immediately.
    • Replacing an existing policy (rotation): goes through the notify + veto window (same machinery as recovery) — a momentary session theft cannot silently swap your guardians. Documented as intended coupling.
  3. GET /v1/auth/recovery/policy (authed) lets the owner review/manage their current (n, k).

Recover — re-bind a passkey after losing the device

  1. POST /v1/auth/passkey/recover/start {account_handle} → issues a single-use server nonce (a PasskeyChallenge-shaped row, operation='recover') and returns {server_nonce, threshold_k}. It does not return approver identities (no social-graph oracle); it reveals only what the login path already does (handle existence).
  2. Client builds the new passkey (WebAuthn attestation) and computes approval_bytes (§5). It asks k guardians out-of-band; each guardian's app verifies who is asking (the social check the gateway cannot make) and signs approval_bytes.
  3. POST /v1/auth/passkey/recover/finish {account_handle, server_nonce, new_credential_attestation, approvals[]} → the gateway: (a) consumes the nonce atomically; (b) verifies the new passkey attestation; (c) recomputes approval_bytes and verifies each approval against a stored approver pubkey, requiring ≥ k distinct approvers from the set (one guardian can't sign k times); (d) opens a pending_recovery row staging the credential with veto_deadline = now + WINDOW; (e) notifies existing devices. Returns {status:'pending', veto_deadline, finalize_token} — no session yet. finalize_token is high-entropy (the ULID recovery_id is ordered, not secret).
  4. POST /v1/auth/passkey/recover/finalize {recovery_id, finalize_token} (client polls) → a single guarded UPDATE gated on the deadline (§7). The winner re-asserts the account is live + in good standing, enrolls the staged credential through the passkey door, revokes old passkeys and old signing keys (clean re-key on the new device), issues a session, and deletes the pending row — one commit. Idempotent: a later poll matches 0 rows and returns the already-issued session.

Cancel — the owner vetoes

POST /v1/auth/passkey/recover/cancel, from an existing authenticated device within the window, deletes the pending row (a guarded delete — §7). A malicious recovery becomes a tap of "No, that wasn't me."

7. Time-locked veto — the state machine (fixes C2)

  recover/finish ──▶ [ PENDING ]  (veto_deadline = now + WINDOW, new cred staged)
                        │  │
      owner cancels ────┘  └──── deadline passes, finalize polled
      (live device)               │
        ▼                         ▼
     [ CANCELLED ]            [ FINALIZED ]  (new passkey bound, old passkeys+signing
     (row deleted;            keys revoked, session issued, pending row deleted)
      old creds intact)

The finalize is not observe-then-write. It is one guarded statement, the deadline folded into the WHERE and arbitrated by rowcount — the codebase's own anti-TOCTOU contract (consume_challenge, _capped_insert):

DELETE FROM pending_recovery
 WHERE user_id = :uid
   AND veto_deadline <= :now          -- server wall-clock (_utcnow), never client-supplied
 RETURNING staged_credential_id, staged_public_key, staged_sign_count
-- rowcount != 1  →  cancelled, absent, or too-early: FAIL CLOSED
-- rowcount == 1  →  this caller won; enroll + revoke + issue session in the SAME txn
  • Cancel wins the race, provably. A cancel that lands first deletes the row → finalize's DELETE matches 0 rows → fail closed. A cancel after finalize finds nothing → no-op. There is no interleaving where both win — the v1 headline claim is now earned, not asserted.
  • Write-once deadline. veto_deadline is set at row birth and never advanced — repeated recover/finish can't push the window out. A second finish while a row is live is rejected; an expired-unfinalized row is reclaimable by a fresh recovery (a guarded UPDATE whose WHERE requires expiry), so the UNIQUE(user_id) slot can't be weaponized into a lockout.
  • One clock. Wall-clock _utcnow() (tz-aware) is the sole deadline clock in both finish and finalize; the finalize payload carries no time field (recall the time.monotonic JWKS bug, #64 — monotonic is wrong for a real-world "24h from now").
  • Notification. Ideal channel is push (#1406, not built); floor is GET /v1/auth/recovery/status. Because a quorum attack is non-silent, the veto is a backstop rather than the sole defense, so the push dependency does not gate v1 — but notification-on-every-recovery is a named requirement, not an afterthought.

8. Data model & migration migration 0013

Three new tables, all FK users.id, all with explicit purge (no ON DELETE CASCADE, per codebase convention):

recovery_policies
  user_id        FK users.id  UNIQUE   -- one active policy per user (rotation replaces)
  threshold_k    Integer               -- quorum size
  created_at     DateTime

recovery_approvers
  id             String(26) PK (ulid)
  user_id        FK users.id  index    -- the account this key can help recover
  approver_pubkey  String(128)         -- multibase Ed25519 (like SigningKey.pubkey)
  label          String  NULL          -- client-side hint only, opaque to the gateway
  created_at     DateTime
  UNIQUE(user_id, approver_pubkey)      -- named constraint (parity gate); enforces distinct approvers

pending_recovery
  id             String(26) PK (ulid)
  user_id        FK users.id  UNIQUE   -- one in-flight recovery per user
  staged_credential_id  String(512)
  staged_public_key     Text           -- base64url(COSE), staged not yet live
  staged_sign_count     BigInteger      -- the attestation baseline (may be stale at finalize — expected)
  veto_deadline  DateTime               -- write-once
  finalize_token_hash   String          -- sha256 of the high-entropy finalize token (never store it raw)
  created_at     DateTime
Three FKs now gate the build (up from two). The #46 cascade guard (test_account_deletion_cascade_guard.EXPECTED_USERS_FK_COLUMNS) will fail until recovery_policies.user_id, recovery_approvers.user_id, and pending_recovery.user_id are each (1) added to the expected set and (2) torn down in accounts_service.delete_user_account via new purge_user_recovery_* helpers (mirroring signing_keys_service.purge_user_keys), children-before-parent. Confirm deletion also invalidates live sessions (finalize is the one flow that mints a session for a not-previously-authenticated party). The test_migrations_match_models parity gate requires the hand-written 0013 to match the ORM exactly (named constraints, not column-level unique=True).

10. What it closes (honest ledger)

  • Self-custody holds end to end. No island is ever a recovery authority; a leaked DB recovers nothing (it has verifier keys, not signing keys).
  • Silent recovery is impossible. An attacker must contact k friends; any one is a warning channel — the structural win over passive key-splitting.
  • The veto is earned, not asserted (guarded UPDATE); cancel-from-live-device provably wins.
  • Residual, named: k genuinely-colluding-or-coerced guardians who all stay silent and act while the owner can't veto can take the account. That is the irreducible k-of-n trust assumption the user chose; the veto + grapevine shrink it but don't erase it. The gateway learns (n, k) + n opaque keys — a reduced, named social-graph exposure.
  • Residual, named — SESSION PLANE (cage-match Tesla, PR#69): finalize revokes the old credentials (passkeys + signing keys) but NOT existing JWT sessions — they're stateless HS256 tokens with no revocation mechanism, so a lost/old device keeps riding its refresh token until it expires (refresh TTL = 30d). Acceptable only because recovery is deploy-dark and un-invokable today. A per-user token_generation revocation mechanism (claude-tasks #1914) MUST land and be live-verified before recovery is enabled for real users — the explicit lift condition.

11. Open questions

  1. Veto window length — 24h / 72h / 7d? Per-island configurable? (The one genuine product-feel dial left.)
  2. Federation portability. The policy lives on one gateway's SQLite; if the island dies, recovery dies (same blast radius as the account itself). Should the policy be a portable, self-authenticating artifact carried on the bus (aligns with key-as-identity, #1569/#1760)? Named as a v2-limitation / federation follow-up, not a v1 blocker.
  3. Guardian-set health. The app should periodically confirm ≥k guardians are still reachable and warn the owner before they silently fall below threshold. A named app requirement.
  4. Bootstrap. A brand-new user with no guardians gets only the printed 1-of-1 key (a bearer secret: lose it → lockout, leak it → takeover, no rotation without re-enroll). Prompt to add guardians once friends arrive?
  5. Rotation & delete (deferred, fail-closed). The gateway currently REFUSES to replace an existing policy (409) and has no policy-delete endpoint — a silent guardian-swap window is never entered, but the owner can't yet change guardians online. Veto-gated rotation + a delete endpoint are a named follow-up.
GO-LIVE GATE (from the PR#69 cage-match). Recovery ships deploy-dark. Before the feature is enabled for real users, claude-tasks #1914 (per-user token_generation session revocation) MUST be live — otherwise a recovered-away device retains access for up to the 30-day refresh TTL (§10 session-plane residual). This is the explicit lift condition for the residual, not an optional enhancement.
