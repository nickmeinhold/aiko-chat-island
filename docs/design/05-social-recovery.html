Social recovery self-custody · guardian-held key shares

The recovery hatch that Design 04 left as an open fork. Decision (Nick, 2026-07-10): self-custody via social recovery — "self-custody but give your friends parts of your key." A recovery secret is split k-of-n across the user's chosen guardians (Shamir); no island or provider is ever a recovery authority. Status: design — for the temper (§9), then build. Grounded on gateway source (domain/models.py, domain/signing_keys_service.py, domain/passkey_service.py, tests/test_account_deletion_cascade_guard.py). Last grounded 2026-07-10.

1. Headline

Recovery is, by construction, an account-takeover primitive: whoever can re-bind a passkey to your account owns it. Social recovery makes the authority to do that a secret you minted and split across friends you chose — never Google, never the island. The design keeps the gateway completely blind to the sharing scheme: it stores one public key (a commitment) and verifies one signature. All Shamir math, guardian delivery, and the printed fallback live in the app. A time-locked veto turns "k friends collude" from silent theft into a cancellable, notified event.

2. What's decided (and what this is not)

Design 04 posed the recovery fork as self-custody (recovery codes) vs. assisted (email/OTP). The answer is a third path that the two-option frame excluded: self-custody, but the recovery secret is socially sharded. Sub-decisions taken 2026-07-10:

QuestionDecision
Recovery authorityA user-minted secret R, Shamir k-of-n across guardians. No provider/island hatch.
TimingTime-locked veto: recovery notifies existing devices and waits out a delay in which the real owner can cancel.
Share deliveryIn-network, each share encrypted to the guardian's aiko identity key (guardians must be aiko users for this path).
Solo fallbackYes — a printed master key for the no-guardians / brand-new-user case. It is the n=1 degenerate share (see §4).
Out of scope. This is not moving auth onto the bus (#1680), not touching sovereign message signing (#1816), and not the password path (/register//login, separable per Design 04 §5). It is one new capability: re-bind a passkey to an existing account by proving possession of that account's recovery secret.

3. Threat model — name the primitive first

Recovery = "enroll a new passkey on account U, revoke the old ones." That is account takeover with the owner's blessing. Enumerate who can drive it and what stops them:

Actor / eventWhat they getWhat stops it
The legit owner, lost deviceRecovers by assembling k shares (or reading the printed card)— (this is the happy path)
k guardians collude, or one guardian coerced k−1 othersCan reconstruct R and initiate recoveryTime-locked veto (§6): the owner's still-live devices are notified and can cancel during the window. Choosing k and n is the user's trust dial.
The island / a leaked gateway DBSees only the commitment pubkey + a pending_recovery rowThe seam (§4): the gateway never holds R or any share. A leaked DB cannot recover an account — same property as the passkey DB holding only public keys.
Network attacker on the recovery requestSees a challenge + a signature over itSignature is over a single-use server challenge (the consume_* pattern); replay is dead on burn. R never crosses the wire.
Account-existence proberWants to learn if a handle/pubkey maps to a userrecover/start returns a challenge unconditionally (§5); no oracle.
Owner with no surviving device (the true lockout)Cannot vetoCorrect — that is exactly when recovery should proceed. The veto defends against malicious recovery while the owner still holds a device; it is not a second factor on legitimate recovery.

4. The seam — the gateway is Shamir-blind

R is an Ed25519 keypair. The private key is Shamir-split across guardians (and/or written on the printed card). The gateway stores only the public key, as a commitment, and verifies a challenge-signature against it. It never learns how many shares exist, who holds them, or how R was reassembled. 
  R = Ed25519 keypair  (minted client-side at enrollment)
  ├─ private key  ─┬─ Shamir k-of-n  →  encrypted to each guardian's identity key
  │                └─ (n=1 case)      →  the printed master card
  └─ public key   ────────────────────→  the ONLY thing the gateway stores

  Recover:  client reassembles R locally  →  signs a server challenge
            gateway checks sig against the stored pubkey  →  never sees R

Why a pubkey commitment, not a hash

If the gateway stored H(R) and the client sent R to prove knowledge, R would cross the wire in plaintext and the island would momentarily be a recovery authority — self-custody broken for that instant. With R as a keypair the proof is a signature over a challenge; the secret stays on the client. This is exactly the property the passkey pivot already bought us (PasskeyCredential stores only public_key; "a leaked DB yields only public keys, which are worthless").

The printed card is the n=1 share, not a second mechanism

The printed master key is just R's private key written down. Reassembling from three guardians and reading it off paper are two client-side ways to obtain the same R. The gateway cannot tell them apart and does not need to — one verification path covers both the social case and the printed fallback. "Yes to printed fallback" costs the gateway nothing extra.

Contrast with SigningKey (#1816), deliberately. The signing-key roster stores a pubkey the gateway never verifies against — it is a carrier, so uniqueness is per-user and a collision is kept as evidence, not rejected. The recovery commitment is the mirror image: proof-of-possession does exist here (the challenge-signature), so the gateway legitimately verifies, the commitment is authoritative and unique per user, and enrollment happens authenticated (no "first to present the key wins" griefing vector, because you must already be logged in to set your recovery pubkey). Same table shape, opposite trust posture — worth stating so a reviewer doesn't pattern-match the wrong invariant.

5. The flows

Enroll (authenticated) — set up recovery

  1. App mints R, Shamir-splits the private key k-of-n, encrypts each share to a guardian's aiko identity key, delivers them, and shows the printed card.
  2. App registers the commitment: POST /v1/auth/recovery/enroll (session-authed) {recovery_pubkey, k, n}. Gateway stores one active recovery_commitment for the user (re-enroll = rotation, replaces the row). k/n are stored for display only — the gateway does not enforce them (it can't; it's Shamir-blind).

Recover — re-bind a passkey after losing the device

  1. POST /v1/auth/passkey/recover/start {recovery_pubkey} → gateway issues a single-use challenge (a PasskeyChallenge-shaped row, operation='recover'), returns it. Always returns a challenge — no account-existence oracle.
  2. Client reassembles R (from k shares or the printed card), signs the challenge, and also runs a normal WebAuthn registration for the new passkey.
  3. POST /v1/auth/passkey/recover/finish {challenge_state, recovery_sig, new_credential_attestation} → gateway: (a) consumes the challenge atomically (the guarded-UPDATE consume_* pattern), (b) finds the user by recovery_commitment.pubkey, (c) verifies recovery_sig against it, (d) verifies the new passkey attestation, (e) opens a pending_recovery row staging the new credential with veto_deadline = now + WINDOW, (f) notifies the user's existing devices. Returns {status:'pending', veto_deadline} — no session yet.
  4. POST /v1/auth/passkey/recover/finalize {recovery_id} (client polls) → if now ≥ veto_deadline and not cancelled: enroll the staged credential, revoke the old passkeys, delete the pending_recovery, issue a session. Idempotent; gated on the deadline server-side, so a client cannot rush it. No scheduler needed (same "self-expire, no sweeper at this scale" stance as OAuthState/SocialNonce).

Cancel — the owner vetoes

POST /v1/auth/passkey/recover/cancel, called from an existing authenticated device within the window → deletes the pending_recovery. A malicious recovery is now a notification the owner taps "No, that wasn't me" on.

6. Time-locked veto — the state machine

  recover/finish ──▶ [ PENDING ]  (veto_deadline = now + WINDOW, new cred staged)
                        │  │
      owner cancels ────┘  └──── deadline passes, finalize polled
      from live device            │
        ▼                         ▼
     [ CANCELLED ]            [ FINALIZED ]  (new passkey bound, old ones revoked,
     (row deleted;            session issued; pending_recovery deleted)
      old passkeys intact)
  • One pending recovery per user — UNIQUE(user_id) on pending_recovery. A second recover/finish while one is pending either no-ops onto the existing row or is rejected (temper decides; lean: reject, so an attacker can't refresh the deadline).
  • Notification degrades gracefully. The ideal channel is push (#1406/#1445), not yet built. Without it the veto still works: an authenticated device sees the pending recovery via GET /v1/auth/recovery/status (or a field on /me) and can cancel on next open. Push only makes the alert timely; it is not a blocker.
  • Window length is a §9 temper call (24h / 72h / 7d). Longer = more chance to catch a malicious recovery, slower legitimate recovery.

7. Data model & migration migration 0013

Two new tables, both FK users.id. Following the codebase convention (no ON DELETE CASCADE; account deletion tears children down explicitly, children-before-parent):

recovery_commitments
  user_id        FK users.id   UNIQUE   -- one active commitment per user (rotation replaces)
  recovery_pubkey  String(128)          -- the Ed25519 pubkey (multibase, like SigningKey.pubkey)
  k, n           Integer                -- display-only (gateway is Shamir-blind)
  created_at     DateTime

pending_recovery
  id             String(26) PK (ulid)
  user_id        FK users.id   UNIQUE   -- one in-flight recovery per user
  staged_credential_id   String(512)    -- the new passkey's credential_id
  staged_public_key      Text           -- base64url(COSE), staged not yet live
  staged_sign_count      BigInteger
  veto_deadline  DateTime
  created_at     DateTime
The cascade guard will fail the build until this is wired — by design (#46, test_account_deletion_cascade_guard). Both new FKs to users.id must be (1) added to EXPECTED_USERS_FK_COLUMNS, (2) torn down in accounts_service.delete_user_account via new purge_user_recovery_commitment / purge_user_pending_recovery helpers (mirroring signing_keys_service.purge_user_keys), children-before-parent. The test_migrations_match_models parity gate also requires the hand-written 0013 to match the ORM exactly (named constraints, not column-level unique=True).

8. What the gateway owns vs. what the app owns

Gateway (this repo — cage-match by law)App (aiko_chat_app — separate PR)
  • recovery_commitments + pending_recovery tables, migration 0013
  • domain/recovery_service.py: enroll (upsert commitment), start (issue challenge), finish (verify sig + stage + open pending), cancel, finalize (deadline-gated, revoke old, issue session)
  • Ed25519 challenge-sig verification (reuse domain/signing.py multikey decode)
  • Cascade-guard extension + purge helpers
  • Rate-limit recover/* (the rate_limit("passkey") bucket already exists)
  • Shamir split / reassemble of R's private key
  • Encrypt each share to a guardian's identity key; in-network delivery + accept/store on the guardian side
  • Printed master-key export (the n=1 share)
  • Guardian-picker + recovery UX; the "recovery pending — cancel?" surface
  • Guardian-set change = re-mint R + re-enroll (a removed guardian may keep their old share)

The seam is clean because the wire contract between them is tiny: a pubkey at enroll, a signature at recover. Everything cryptographically interesting is client-side.

9. Open questions for the temper

  1. Fresh R vs. reuse the sovereign signing key (#1816)? Lean fresh: the signing key is hot (on-device, used every message); the recovery key is cold (split, offline, rarely touched). Splitting your hot signing key among friends would be a category error. But confirm — reuse would mean one fewer keypair for the user to understand.
  2. Veto window length — 24h / 72h / 7d? And is it per-island configurable?
  3. Notification path before push exists. Ship GET /recovery/status now and treat push as a later upgrade, or hold the veto UX until #1406? (Lean: ship status-poll now, it's the graceful-degrade floor.)
  4. Does finalize revoke old signing keys too, not just passkeys? Recovery implies "my old device is gone / possibly compromised." Lean: revoke old passkey_credentials and old signing_keys, forcing a clean re-key on the new device.
  5. Force re-enroll of R after a successful recovery? The shares that were just assembled are now "spent" (guardians saw them travel). Lean: yes — prompt a fresh split post-recovery.
  6. Concurrent second recover/finish while one is pending — reject (don't let an attacker refresh the deadline) vs. no-op onto the existing row?

10. Build plan

  1. Temper this doc (design cage-match, ~2 rounds — trust boundary, but 0 real users + undeployed, so calibrated light). Settle §9.
  2. Gateway PR (cage-match by law): migration 0013, recovery_service.py, the four recover/* endpoints + enroll + status, cascade-guard extension, tests (RED-prove: a leaked commitment alone can't recover; a replayed challenge fails; finalize before deadline fails; cancel from a live device wins the race).
