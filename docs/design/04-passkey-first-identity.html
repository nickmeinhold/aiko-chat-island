Passkey-first identity retiring social sign-in

Proposal to drop federated social sign-in (Apple / Google / GitHub OAuth) and make a device-held passkey the identity root, with a deliberate recovery hatch. Motivated by the passkey-401 bug (passkey sign-in returns 401 because passkeys are bolted onto the social account flow) and by coherence with the sovereign-signing work just shipped (gateway #66/#67). Status: design — pending team vote on the recovery fork (§7). Grounded on the gateway repo source (rest/auth.py, domain/passkey_service.py, domain/users_service.py, domain/models.py) and a mid-2026 capability review (§4, cited). Last grounded 2026-07-10.

1. Headline

The instinct "get rid of social accounts" is sound: social OAuth is the one centralized Apple/Google dependency that contradicts a self-hosted, federated island. But the evidence (§4) is blunt — no credible service ships auth with zero fallback. So the decision is not "passkey-only vs. keep social." It is: remove social OAuth, make the passkey the identity root, and add a thin recovery hatch that is NOT a federated identity. The only open question is which hatch (§7). Critically, the hard primitives already exist in the codebase — this is mostly subtraction plus two small additions.

2. What's true today

Read from source, not from comments:

  • Three sign-in kinds coexist: native social (Apple/Google id-token → /v1/auth/social), broker social (GitHub server-side code flow → /oauth/*), and passkey (WebAuthn → /v1/auth/passkey/*). Plus a dev-only password path (/register+/login, open_registration=False in prod).
  • A passkey does not persist at registration. passkey/register/finish verifies the attestation, packs the credential into a gateway-signed provisioning token, and returns it. The credential is only written when the app calls the shared /social/claim endpoint with a chosen handle (passkey_service.py:13, rest/auth.py:302).
  • This is the passkey-401 bug: if the handle is already taken — e.g. you hold "nick" via an existing social account — the claim is rejected, the provisioning token is discarded, and the device is left holding a passkey with no server-side row. Every later authenticate/finish then correctly reports "unknown credential" → 401, forever.
  • The standalone-persist primitives already exist: /passkey/add/finish (rest/auth.py:429) persists a credential directly against an authenticated user via link_passkey_credential — no token, no claim. And create_passkey_user (users_service.py:98) mints a User + PasskeyCredential atomically. Both are reachable today only through the colliding social-claim detour.
  • The User row: username (unique), aiko_username (unique, bus attribution), password_hash (nullable — social & passkey accounts carry None), email (nullable). A passkey account needs no password and no third-party token.

3. The reframe

It is not "passkey-only vs. keep social." A recovery hatch (recovery codes, or email/OTP) carries zero Google-identity dependency — it is a backup/delivery channel, not a federated identity. Removing social OAuth and adding a recovery hatch is fully consistent with the sovereignty thesis and with the evidence that fallback-less auth is an outlier bet. The passkey becomes the identity root — which is exactly what the sovereign-signing work (a device-held keypair bound to an account, #66/#67) already assumes. Social login was always the awkward centralized graft.

5. The design (3 shippable steps, in order)

Step 1 — Passkey creates its own account this IS the #1728 fix

Replace the provisioning-token/claim detour: register/finish persists a fresh account immediately — a User with an auto-generated unique handle (aiko-a3f9c1, exactly the provisional name passkey_service.py:151 already mints) plus the PasskeyCredential, atomically, via the same direct-persist that add/finish already proves. The unlocking insight: once you authenticate by passkey, the username is cosmetic, not a login credential — it needs no pick-and-confirm before the account exists. Renaming becomes a profile edit, never an account-creation gate. This deletes the entire provisioning-token dance and closes #1728.

Step 2 — Recovery codes (the escape hatch the evidence demands)

A new single-use-token table, mirroring the exact consume pattern the codebase already uses everywhere (consume_challenge / consume_nonce / consume_state). Details in §6.

Step 3 — Delete the OAuth/social stack the subtraction win

Removed entirelyTrimmed
oauth.py, oauth_broker.py, handoff_service.py, nonce_service.py, pkce.py, state_service.py; SocialIdentity model + oauth_states/social_nonces tables; the entire provisioning-token machinery (security.issue/decode_provisioning) — nothing uses it once passkey persists directly. rest/auth.py: /social, /social/claim, /nonce, /oauth/{start,callback,exchange}; users_service: create_social_user, get_user_by_social; config.py: apple_client_ids, google_client_ids, broker config, social_nonce_required, social_signin_enabled; /providers social entries.

~1,000–1,500 LOC of the highest-risk code in the gateway, and it dissolves (not just closes) whole classes of open work — app-authenticity for the OAuth handoff, the social-nonce replay-defense flip, and the broader replay-defense lineage. Each step is independently shippable and cage-matched (trust boundary → cage-match by law).

6. Recovery codes Step 2 detail

recovery_code:
  id          ULID  PK
  user_id     FK users.id, index
  code_hash   String, unique per user   # store ONLY sha256/argon2 of the code
  used_at     DateTime nullable          # tombstone; NULL = still valid
  created_at  DateTime
  • At account creation: generate 8 codes, show them once, store only hashes (never persist the plaintext — the redaction lesson).
  • POST /v1/auth/passkey/recover: submit a code → atomic conditional UPDATE … WHERE user_id=? AND code_hash=? AND used_at IS NULL, arbitrated by rowcount (TOCTOU-free, idiomatic here) → on rowcount==1, issue a session and route straight into add/finish to enroll a passkey on the new device. Code burned atomically with the outcome (the defer-commit contract, #24).
  • Threat framing: a recovery code is a bearer credential — same model as a crypto-wallet seed phrase. That is the self-custody bargain, and it is the honest thing to tell users. It is not a federated identity, so it does not reintroduce the dependency social OAuth carried.

7. The fork (team call)

The recovery hatch maps directly onto who aiko is for. This is the decision the team owns.
Recovery codes only (self-custody)+ risk-scored email/OTP fallback (assisted)
FitsThe sovereignty ethos. Zero PII, zero external dependency, no new infra.Reach. Catches both bleed points (§4) — the iOS→Android switcher and the de-Googled/China user.
CostsDoes nothing for the two bleed points. A lost set of codes = lost account (accepted as principled).Reintroduces email as a channel (PII + an SMTP/SES dependency the gateway lacks today). Far lighter centralization than OAuth, and per-island-optional keeps it sovereign.

8. Gotchas to design for now

1. The cascade guard will (correctly) fail the build until recovery_codes is added to the account-deletion purge. The #46 Base.metadata-introspection tripwire trips on any new users.id FK unless you add purge_user_recovery_codes + extend the EXPECTED FK set. That is the runtime guard earning its keep — a feature, not friction.
2. Existing social accounts get stranded. The 2 Sydney test accounts are password_hash=None + SocialIdentity, no passkey — deleting social locks them out. At 0 real users / 2 test accounts this is "wipe and re-register" — which is exactly why now is the cheap moment. For a real migration with real users you'd need a "link a passkey before we remove social" window; that door closes cheaply only while it's empty.
3. De-Googled Android is a primary-enrollment problem, not just recovery. A Huawei user can't create a synced platform passkey locally; they need browser-WebAuthn + cross-device QR to work at all. Independent of the recovery fork, and worth scoping if the de-Googled audience matters.

Not bundled: the dev-only password path (/register+/login) is separable. "Kill social" (OAuth) ≠ "kill passwords"; treat the password path as a later, separate call.

9. Recommendation

Do the core move: remove social OAuth, go passkey-first, via the three steps above (Step 1 is unavoidable — it's the #1728 fix — and makes passkey-only viable; Step 2 makes it safe; Step 3 is the subtraction win). Do not ship it fallback-less. Recovery codes are the minimum viable hatch and fit the ethos — a defensible baseline. But the evidence says recovery-codes-alone will bleed the cross-device switcher and exclude de-Googled/China; a per-island-optional email/OTP hatch is the evidence-backed way to plug that without the centralization social OAuth carried. The self-custody-vs-assisted choice (§7) is the team's, and it is really a choice about audience.
