Auth on the Bus where auth belongs

Issue #1680 asked "move auth to the bus: passkey → JWT → gate the MQTT connection." Grounding in the live code shows the issue fuses two very different questions. This doc separates them, names the real federation threat, and marks the one fork that gates everything. Status: design — Layer A buildable now; Layer B awaiting the users Category in ChatServer + one federation fork. Grounded on the gateway repo source and, for the upstream claims, on Andy Gelme's Signal answers of 2026-07-04 verified against geekscape/aiko_chat source (chat.py), not from issue prose. Last grounded 2026-07-05.

1. Headline

The issue's framing — "gate the MQTT connection with a passkey-derived JWT" — pictures end users authenticating to the bus. They don't. Users talk HTTP/WS to the gateway; the gateway is the sole bus participant on behalf of every user. So "auth on the bus" splits into two independent layers: (A) service-to-broker auth (close allow_anonymous — buildable now) and (B) user-identity-as-bus-peer (Andy's native model — needs the architecture to move users onto the bus, and converges with the federation-identity work). The threat that actually motivates either one is mesh-to-mesh, not user-to-bus — and it doesn't exist yet.

2. What's actually true today

Read from source, not compose comments (the config has been wrong 3× on this repo):

  • deploy/mosquitto.conf: allow_anonymous true, no password_file, no acl_file.
  • docker-compose.yml: the broker publishes no host port — it is reachable only within the compose network. The gateway binds 127.0.0.1:8095; Caddy terminates TLS. So anonymous is genuinely safe today (the conf's own comment says: "If a future island ever exposes the broker off-host, this MUST gain auth first").
  • The bus has exactly three clients, all co-located trusted containers: the gateway (aiko/client.py → paho, an ECConsumer on channel_list), the registrar, and the ChatServer.
  • The app never touches MQTT. Its entire surface is rest/* + realtime/ws.py (HTTP/WebSocket). The gateway proxies onto a bus it is the only citizen of.
  • The JWT is HS256 symmetric (domain/security.py, one shared jwt_secret).
  • config.py exports only AIKO_MQTT_HOST/_PORT — no MQTT credential path exists in code today.

3. The reframe: two layers, not one

Layer A — service-to-broker authLayer B — user-identity-as-bus-peer
Who authenticatesThe gateway / registrar / ChatServer authenticate to mosquitto.Each user authenticates to the bus with their own key ("I'm Nick, here's my public key"), then per-topic access controllers.
What it closesallow_anonymous. Three trusted containers present credentials (password_file or mTLS).The gateway's HTTP auth layer as the sole security boundary — the "temporary scaffold" the issue smells.
PreconditionNone — buildable now. No user identity involved.The architecture must move users onto the bus as peers (the app stops being an HTTP client). A P2P-vs-client-server move, not JWT plumbing.
Whose modelOurs (ordinary broker auth).Andy's native aiko model (see §6). We align, we don't reinvent.

4. The real threat is mesh-to-mesh, not user-to-bus

Today the bus is a sealed cul-de-sac (no host port). It only becomes exposed when islands federate — and the exposure is an inter-mesh link between gateways/brokers, not end users hitting an open broker. So the thing to secure first is the federation edge between islands. That edge does not exist yet, which is exactly why Layer A is "buildable but not yet load-bearing" and Layer B is "north star, not yet scheduled."

5. Two findings that fell out of grounding

Finding 1 — the decouple (#1685) shipped without the broker auth its own spec required. #1685's build-step 2 says "ship password_file + acl_file in the repo, deployed from the repo (substrate-first)." The shipped mosquitto.conf is allow_anonymous true. So Layer A is really unfinished #1685 scope — deferred defensibly (internal-only), but the substrate-first intent was silently dropped. A named-but-unfixed tradeoff sitting in a file we own.
Finding 2 — HS256 is island-local by construction. A shared symmetric secret can't verify cross-island tokens without sharing the secret between independent operators — a non-starter. Federation-era identity needs asymmetric keys (RS256/EdDSA + JWKS) or DID-style key-as-identity. That is the same crypto decision as Design 09 (#1569) "key-as-identity, reputation-not-personhood." Layer B and the sybil/identity work converge on one keypair choice.

6. Andy's model gives Layer B a concrete home

Andy Gelme's OKF documentation (2026-07-04) and his 318-Russell-St model both put auth at the HyperSpace / Category function-call level: authenticate to MQTT at connect with a public key, then per-topic access controllers; the HyperSpace roadmap explicitly lists "security (owners, roles, ACLs)," and every Dependency/ServiceFilter already carries an owner field. That is finer-grained than "JWT-gate the connection" — per-entry ownership + per-operation ACL, checked at the Actor. It is Layer B's native shape. See reference_hyperspace_api_vs_ec_interaction and reference_aiko_architecture_andy.

The concrete write-path surface this rests on — Category.add/list/update/remove, entries-are-Dependency, the depth-2 share limit, and the EC-control-topic trap — is pinned in Bus Interaction Contract (grounded on aiko_services v0.7 concepts/, 2026-07-08). The owner field Layer B attaches to is already present on every entry there.

Grounding gate (sharpened 2026-07-05): Layer B has two concrete upstream blockers, not a vague "roadmap." (1) The users Category itself is not built in ChatServer — a TODO comment (chat.py:46,49), so there is no user identity in HyperSpace to attach ownership/ACLs to yet (see §2b). (2) The HyperSpace security model (owners, roles, ACLs) is v0.6 WIP — update() unimplemented, ACLs planned, remote create() has a proxy bug. Do not build Layer B ahead of both landing. Layer B = align with a maturing upstream model, on Andy's timeline — the first observable milestone is his promised example code standing up a populated users Category a remote client can list.

7. The fork (the decision that gates everything)

The federation threat is mesh-to-mesh, and it doesn't exist yet — so the load-bearing question is how islands federate. The original framing was a two-item menu; Andy's "HyperSpace mounts like a disk into a root HyperSpace" (§2b) is a third option the menu excluded, and it may be the native one.

SubstrateHow A talks to BConsequence for auth
1. MQTT-broker bridge Broker-to-broker link; users eventually become bus peers. Layer B becomes reachable; asymmetric key-at-connect + per-topic ACLs (Andy's model). Broker exposed off-host → Layer A becomes load-bearing first.
2. Gateway HTTP federation API A REST/federation edge between gateways; each bus stays island-internal. Bus stays a sealed cul-de-sac; Layer A never has to ship; asymmetric keys needed only at the HTTP edge for cross-island token verification.
3. HyperSpace mount into a root graph Andy's native model Each island's embedded HyperSpace mounts into a larger "root" HyperSpace; you traverse the whole mesh with one API and one set of semantics. Auth lives where Andy already puts it — per-Dependency owner + Category ACLs, checked at the Actor, uniformly across the mounted graph. Subsumes Layer B rather than bolting federation on top. Most aligned, least mature (needs the v0.6 security model + the mount semantics proven remote).
Everything downstream — whether Layer B is even on the roadmap, whether we migrate JWT to asymmetric now, whether the gateway stays the user-facing surface or dissolves into a peer — hangs off which substrate we bet on. This is Nick's call, and it wants a decision, not more elegant downstream refinement. My read: option 3 is the north star but blocked on upstream maturity; option 2 is the pragmatic near-term path that keeps the gateway sovereign; option 1 is the awkward middle. Recommend deciding 2-vs-3 explicitly rather than drifting.

8. Recommendation

  • Layer A — do it, as the honest completion of #1685. Small substrate-first PR: repo-committed password_file + acl_file, teach config.py/aiko to pass MQTT creds, flip allow_anonymous false. It retires a standing named tradeoff and is a proactive default even though the bus is still internal-only. Worth doing with (or just before) the first federation-bus work.
  • Layer B — align, don't reinvent; hold for two concrete milestones. It's the north star and converges with Design 09 (#1569) on asymmetric key-as-identity, but it's blocked on (i) ChatServer populating a users Category a remote client can list (today it's a TODO comment, §2b), (ii) the HyperSpace security model maturing past v0.6, and (iii) the federation-substrate fork (§7) resolving toward option 1 or 3. Andy's promised example code is the first observable signal on (i)/(ii).
  • Migrate JWT HS256 → asymmetric only once the fork resolves toward cross-island token verification (don't pay the complexity before federation needs it — anticipate the problem, let measurement/decision pick the fix).
