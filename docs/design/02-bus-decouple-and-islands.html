Bus Decouple & Islands islands prereq-0

The gateway currently borrows its message bus from the matrix bridge stack. This unentangles it onto its own aiko mesh — the precondition for a portable "island". 1. The headline (read this first)

What's wrong: aiko-chat-gateway attaches to the matrix project's matrix_default network and points AIKO_MQTT_HOST=aiko-mosquitto at matrix-aiko-mosquitto-1 — a broker owned by the matrix compose project. Tear down the matrix stack and the gateway's bus (and the ChatServer it reads channel topology from) goes with it.

What changes: give the gateway its own aiko mesh — mosquitto + registrar + ChatServer — in its own compose project on its own network. The mesh definition already exists (aiko-chat-bridge/deploy/docker-compose.aiko.yml); we take it minus the aiko-bridge (matrix-connector) service.

Why it's prereq-0 for islands: the decoupled compose is the island template. Island #2 = same compose, different host + domain + JWT_SECRET. You can't make the bus portable while another project owns it.

This is the runtime-state-machine sibling of remove-the-coupling-not-guard-the-window: we don't add a guard around the shared broker, we delete the sharing.

The bus uses the registrar pattern (the only retained topic on the broker is aiko/service/registrar): services register there, the gateway discovers the ChatServer via it, then ECConsumer(filter="channel_list") reads canonical channel topology from the ChatServer. The key correction to the original handoff: the gateway depends on the ChatServer (matrix-aiko-chat-1), not the matrix bridge. The bridge only injects matrix-originated messages — a message source, never topology.

3. What an island really is

The handoff framed an island as "gateway + own mosquitto + aiko services". The bus map sharpens it:

An island = gateway + mosquitto + registrar + aiko-chat (ChatServer). No matrix, no continuwuity, no mautrix bridges, no aiko-bridge. A clean standalone aiko mesh — which is simpler than today's entangled host and is exactly the "better than Matrix" shape.

Buildability — resolved. All three non-broker roles build from one image, aiko-bridge:latest, whose build context is ./aiko-chat-bridge = github.com/nickmeinhold/aiko-chat-bridge — your own repo. /opt/aiko_chat (geekscape/aiko_chat, the ChatServer code) is baked into that image. So a fresh island host needs only: clone your bridge repo, docker compose build. No Andy dependency, no registry required (a registry would be a nicety, not a blocker).

5. Why swapping the ChatServer is data-safe

Natural fear: the gateway has persisted message history (SQLite is the sole store for messages); does pointing it at a fresh ChatServer orphan that history? No — verified in the code:

  • The gateway keys channels by name, not by a ChatServer-minted id. Channel.id is a gateway-local ULID; upsert_channel matches on Channel.aiko_channel == name and returns any existing row untouched (channels_service.py).
  • So the gateway — not the ChatServer — owns channel identity and message history. A new ChatServer publishing the same channel names re-attaches to the existing rows by name.

Condition: the new mesh's chat_start.sh must bootstrap the same channel names. It's idempotent and seeds the same defaults, so this holds — but it's the thing to verify on the new mesh before repointing the live gateway.

The one real hazard: if the new ChatServer bootstraps a smaller channel set, the reconcile worker could receive a live-producer EC remove for the missing channels and cascade-delete their messages (hard_delete_channel). Mitigation is belt-and-suspenders: (1) same-name bootstrap so no remove fires; (2) the standard backup-first deploy (online SQLite backup to ~/aiko-db-backups/ immediately pre-cutover) so a mistake is recoverable, per the #17 restore drill.

6. Decouple plan + cutover

Reversible prep is done ahead of time; the single irreversible step (repoint + restart) is foregrounded and verified. Never fire-and-forget — this is a live bus migration on your host.

Prep (no impact on the running gateway)

  1. Vendor the aiko mesh into the gateway project: add aiko-mosquitto, aiko-registrar, aiko-chat to aiko-chat-gateway/docker-compose.yml (from docker-compose.aiko.yml, dropping aiko-bridge), plus a project-owned network and the mosquitto.conf. Make matrix_default no longer required.
  2. Ensure the bridge repo + image are available on the host so the gateway project can build the mesh image (or pull a pinned one).
  3. Bring the new mesh up in isolation (its own network, distinct project) and verify on it, WITHOUT touching the gateway: registrar retained topic present; ChatServer registers; chat_start.sh bootstraps the same channel name set as the live mesh (diff the two channel lists — this is the §5 safety condition).

Cutover (foregrounded, one step, verified)

  1. Backup first: online SQLite backup of the live gateway DB to ~/aiko-db-backups/aiko.db.predeploy-<ts> (the #17 drill path).
  2. Repoint the gateway at the new mesh (same project network; AIKO_MQTT_HOST → the new broker) and recreate just the gateway container.
  3. Verify before declaring done: gateway boots; /health 200; reconcile attaches to the new ChatServer; channel set unchanged (no spurious deletes — check row counts against the backup); a round-trip message persists.
  4. Detach the gateway from matrix_default once confirmed independent.

The matrix project is left entirely alone under Option A — its own aiko mesh keeps running for the matrix bridge.
