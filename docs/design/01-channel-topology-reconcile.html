Channel Topology Reconcile #1281 incr 2

Gateway reads canonical channel topology from aiko's HyperSpace and reconciles it into local Channel rows — killing the hardcoded ["general"] drift. Channels only; users deferred (task #9). Status: design, awaiting sign-off.

1. The headline (read this first)

What changes: the gateway stops independently seeding channels (main.py:74-83) and instead mirrors aiko's canonical channel set, received as a live push-stream.

How: attach an ECConsumer(filter="channel_list") to the ChatServer the gateway already discovers. Zero edits to Andy's aiko_chat repo. Runtime-proven 2026-06-25 (see §3).

The two decisions that needed making (both resolved below): (a) how a hard-delete actually deletes given the FK schema, and (b) how to tell a real channel removal from a transient ChatServer blip so we never nuke history by accident.

2. Why this exists — the drift, made concrete

This is a live instance of the project's signature bug, concept_two_representations_drift. Two representations of "which channels exist" disagree:

RepresentationHoldsSource
Gateway (hardcoded)["general"]client.py:43, config.py:46
aiko HyperSpace (canonical)general, random, llm, robot, yolospike/devstack.sh:44-52

The gateway is structurally blind to 4 of 5 channels. The #1281 decomposition makes HyperSpace the single source of truth for channel/user existence; this increment delivers the channel half.

3. Mechanism (proven, not proposed)

aiko's "event-change" (EC) share system: an actor publishes a share dict; remote actors consume it via ECConsumer, which keeps a live local mirror updated by add/remove/update commands. ChatServer publishes self.share["channel_list"] (aiko_chat/chat.py:274), tagged ec=true.

  • The gateway's existing _discovery_add(service_details, service) already receives service_details[0] = ChatServer's topic_path — the exact input ECConsumer needs.
  • filter="channel_list" ⇒ the producer's _filter_compare (share.py:245) returns only channel_list + channel_list.*.
  • Breadcrumb: a commented-out # filter="channel_list" at aiko_chat/chat.py:215 — the aiko authors anticipated this consumer.
Runtime proof (2026-06-25, spike/probe_channel_list.py) 
[probe] discovered ChatServer at aiko/MacBookPro.lan/91011/1; attached ECConsumer(filter='channel_list')
[probe] share event: add channel_list.general = [['*', 'general', '*', '*', '*', []], 'None', 'None']
[probe] share event: add channel_list.llm     = ...
[probe] share event: add channel_list.random  = ...
[probe] share event: add channel_list.robot   = ...
[probe] share event: add channel_list.yolo    = ...
[probe] PASS: all 5 bootstrapped channels arrived via the EC share.

Payload shape (drives the parse): each cache entry under channel_list.<name> is [['*', '<name>', '*', '*', '*', []], 'None', 'None'] — a ServiceFilter-shaped tuple. Channel name = entry[0][1], redundantly the cache-key suffix. Consumer cache = {"channel_list": {"<name>": <tuple>, ...}}.

4. Reconcile semantics

4.1 Name → Channel fields

The share carries only a bare name. Mapping for an upserted row: name=X, aiko_channel=X, kind="standard" (the llm/robot channels could later map to richer kinds — out of scope here), is_private=False (HyperSpace channels are public existence; private/ACL stays a gateway-local overlay per #1281). This matches today's _seed_channels mapping exactly, so the upsert is behaviour-preserving for general.

5. Decision A — how hard-delete actually deletes resolved

"Hard-delete the channel row" is not one behaviour with the current schema. Message.channel_id and Membership.channel_id are ForeignKey("channels.id") with no ondelete clause (domain/models.py), and db.py sets no SQLite foreign_keys pragma. So a naive DELETE channels WHERE id=… gives:
  • Postgres (FK enforced, no cascade) → IntegrityError, delete blocked while messages exist.
  • SQLite (pragma off by default) → delete succeeds, messages orphaned with a dangling channel_id; channel-joined queries break.
Neither is what was chosen.
Resolution: application-level explicit cascade in one transaction — DELETE memberships WHERE channel_id=…, DELETE messages WHERE channel_id=…, then DELETE channels WHERE id=…. Backend-agnostic (doesn't depend on FK enforcement differing between SQLite/Postgres), explicit, and testable. This honours the chosen semantics: a channel absent upstream ceases to exist, history included. (Alternative — ondelete="CASCADE" + enabling the SQLite pragma — is rejected: it couples correctness to a per-connection pragma and a migration, and hides the destruction inside the FK engine.)

6. Decision B — real removal vs transient blip resolved

Hard-delete + cascade is irreversible mass deletion. If a transient event (ChatServer restart, registrar hiccup) emitted a spurious remove, a momentary network blip would wipe a channel's entire history. This is the destructive analogue of the transient-vs-terminal auth distinction (feedback_auth_error_boundary).
Resolution: delete ONLY on a live-producer EC remove event; NEVER on _discovery_remove. Reading the consumer resync path (share.py): a producer restart re-issues add commands (it does not clear-then-remove), so a restart does not synthesise spurious removes — it re-adds. The genuine remove only fires on real category removal while ChatServer is live. Tying destruction to that signal alone — and explicitly not to ChatServer disconnect — bounds the blast radius to genuine upstream removals.
Validation gate — RUN & PASSED (2026-06-25, spike/probe_restart_removes.py). A ChatServer restart under a live consumer produced: discovery_remove (no db-remove) → discovery_add with a new topic_path (new PID) → 5 re-adds. Zero reconcile remove events. Mechanism: restart ⇒ new PID ⇒ new topic_path; the old consumer goes silent against its dead producer, the new consumer gets fresh adds. A restart cannot trigger a hard-delete.
Stronger finding — the EC remove signal cannot fire e2e with current aiko_chat. ChatServer reads channels_list once at init (chat.py:272-274) and never mutates it; there is no runtime add_channel/remove_channel path (dynamic channels are an aiko_chat TODO, chat.py:45-50). So the channel set is fixed at ChatServer startup. Consequence: the hard-delete branch (§5) is defensive / forward-facing — correct for when upstream gains channel removal, but exercisable today only by a synthetic remove in a unit test, never end-to-end. The live value of this increment is the add-reconcile (gateway learns all 5 channels, not 1). See §10 for the resulting build-scope decision.

7. Scope boundary

Channels only. ChatServer publishes channel_list but no user_list share. User topology read-through needs an upstream change to Andy's aiko_chat (task #9). Gateway-local concerns are untouched: auth/credentials, message persistence, and ACL/membership remain gateway-owned overlays per the #1281 decomposition.

8. Test plan (ATDD)

  1. Parse: channel_list payload tuple → channel name (entry[0][1]); tolerate the redundant key-suffix form.
  2. Add reconcile: add events for 5 channels ⇒ 5 Channel rows, idempotent on repeat (upsert, no dupes).
  3. Remove reconcile (live): a live EC remove for X ⇒ channel + its messages + memberships gone (application-cascade), in one transaction.
  4. Transient safety: _discovery_remove ⇒ rows untouched; consumer torn down.
  5. Seed removal: with reconcile active, the hardcoded _seed_channels path is deleted; general still present (now via reconcile, not seeding).
  6. Live e2e (extends e2e/test_bus_roundtrip.py): real devstack, assert all 5 bootstrapped channels reconcile into the DB; the §6 restart validation gate.

10. Build-scope decision (post-spike) awaiting sign-off

The spikes proved the add-reconcile works and that EC removes cannot fire e2e with current aiko_chat. Two viable build scopes:

Scope A — add-reconcile onlyScope B — add + defensive hard-delete
Ships nowGateway learns all 5 channels; delete _seed_channels; ignore discovery-remove (the safety scaffolding stays)Scope A + the application-cascade hard-delete on live EC remove
Hard-delete branchDeferred to a task, gated on aiko_chat emitting channel removalsBuilt now, unit-tested with a synthetic remove (no e2e coverage possible)
Complexity / blast radiusLow — additive, no destructive pathHigher — irreversible cascade behind a branch that can't fire e2e yet

Recommendation: Scope A. Deliver the live drift-kill (5 channels), keep the discovery-remove safety behaviour, and defer the destructive cascade until upstream can actually emit a removal to test against — building an irreversible path we can't exercise e2e is the kind of untested destructive code best deferred until it's reachable. Hard-delete decision (cascade mechanism) is preserved in this doc for when it lands.

9. Review regime

State-lifecycle + destructive persistence ⇒ 3-way cage-match (Maxwell + Kelvin + Carnot) before merge, per the project's per-component regime and the "trust boundaries are cage-match by law" rule. Decisions A and B are the adversarial focus.
