A six-scene explainer

An island of one’s own

What a self-run island actually is — and what one command does to make one: it looks at your host, builds only what’s missing, and insists on owning the few things that make it sovereign.

Scroll to explore
Scene 01

Four things on one patch of land

The idea

Start with what an island actually is. A running chat community needs only four working parts. The pipes that carry messages between people. A directory that says who’s who. A keeper of which rooms exist. And a doorway the app knocks on. Put all four on one patch of land, answering to no one else — that patch of land is an island.

The whole idea in one word: an island is broker + registrar + chatserver + gateway, self-contained. Nothing borrowed, nothing shared. One island runs one community, start to finish, on its own.
The real machinerygateway + mosquitto (MQTT broker) + registrar + ChatServer — one docker compose project.
Scene 02

One command that looks before it builds

The idea

You make one with a single command. But it isn’t a bulldozer — the first thing it does is look. Is there already a store of past conversations here? Leave it exactly where it is. Is there already a secret key holding everyone’s sessions open? Reuse it — replacing it would sign everyone out. It builds only the pieces that are genuinely missing.

Why that matters: you can run it again tomorrow and nothing precious gets clobbered. Look first, build only the gaps — so setting up an island, or repairing one, is the same safe command either way.
The real machinerydeploy/standup.sh — reconciles by compose project + service name; reuses the existing aiko_data volume and JWT_SECRET, builds only what’s missing.
Scene 03

What it borrows, and what it refuses to share

The idea

Now the sharp part. Some things on a host are fine to share — the front gate that handles web traffic and certificates. If the host already runs one, the island steps aside and uses it. But the message bus — the broker — it builds its own, every single time, even if one’s already running right there.

Why the double standard: a front gate is generic plumbing; sharing it costs nothing. But a shared bus is a shared fate — the moment two islands talk through one broker, neither is truly its own. The broker looks like something you’d reuse. Its stubborn separateness is the island.
The real machineryReuses a host reverse proxy (Caddy on :80/:443); always runs its own mosquitto — a shared broker is a shared fate.
Scene 04

Now make another one

The idea

Here’s the payoff for doing it this way. Once an island owes nothing to anyone, you can build a second one just by copying the first. Same design, new patch of sea — a different home, a different name, its own secret key. Two islands, run by different people, who’ve never had to ask each other’s permission to exist.

This is federation: not one giant service everyone crowds onto, but a growing archipelago of small, sovereign ones. The first island was the hard part. Every island after it is a copy with a new flag. chat.imagineering.cc and chat.enspyr.co are exactly this — already live.
The real machineryFederation = N sovereign islands — each its own compose project, domain, and JWT_SECRET (chat.imagineering.cc + chat.enspyr.co are live).
Scene 05

The island remembers on its own

The idea

One worry haunts a setup like this. The keeper-of-rooms can be swapped — rebuilt, replaced, restarted from scratch. When it is, does the island’s memory go with it — every past conversation orphaned? No. Because the island keeps its own logbook, filed by the name of each room, not by some number a keeper happened to hand out.

Why it survives: swap the keeper of rooms, and the new one simply announces the same room names — which snap straight back onto the island’s existing records. The history was always the island’s own to hold. Keyed by name is what makes the island’s memory portable.
The real machineryChannels keyed by name in gateway-local SQLite — swap the ChatServer and history re-attaches by name.
Scene 06

The honest cost: two seas

The idea

Every real design pays for what it gains — so here’s the bill. Because each island is genuinely its own world, they don’t automatically share one. A room on this island isn’t a room on that one; a name here isn’t the same name there. True independence and a single global soup are opposites — you can’t have both.

No pretending otherwise: federation isn’t one giant service wearing many hats. It’s many small worlds that stay separate unless they deliberately reach across. Whatever crosses between islands crosses on purpose — which is exactly the problem the next notes pick up.
The real machineryTwo islands = two channel / message namespaces — crossing is explicit federation, not a shared backbone.

The whole thing, in one breath

An island isn’t a wall. It’s a thing that owes no one its foundations.

Own your four pieces, keep your own memory, share only what you choose — and suddenly the hardest thing in the system is also the most copyable. One island was the invention. A nation of them is just addition.

The one honest cost — two islands are two worlds — isn’t a bug to hide. It’s the shape of sovereignty itself. Whatever wants to cross between islands crosses on purpose, out in the open, by a bridge you can see.
